Practice on your own data

Run the AI Analyst workflow on your data

This repo turns Claude Code into a disciplined analyst that takes a business question through a 7-phase, checkpoint-gated pipeline and produces an HTML report. To practice on your own data, you swap in your schema, metrics, and business context — built in the same structure as the workshop's — then run the exact same workflow. This guide gets you there in seven steps.

⏱ ~30–45 min setup 🗄 Any SQL database 🤖 Claude drafts the files for you 🔁 Same workflow we use live
🗄️
Your dataConnect via MCP
📄
Your contextschema · metrics · business
⚙️
7-phase workflowQA → … → report
📊
HTML reportFindings + verdicts
💡
The one idea to hold onto: the workflow never invents schema or metric logic — it reads them from two files (schema.yml + metrics.yml) and your plain-language business context. Get those three right for your data and the whole pipeline just works.

Connect your data

Point Claude Code at your database via MCP

The workflow reads data through an MCP server — a connector that lets Claude Code query your database directly. Use the MCP that matches your warehouse. You only do this once.

Postgres / Supabase

Use the Supabase MCP or a generic Postgres MCP. Supply your connection string / project credentials.

BigQuery, Snowflake, etc.

Use the MCP for your warehouse. The workflow is SQL-agnostic — it adapts to your dialect from schema.yml.

Add the MCP server

terminal · example (Postgres)
# Example — add a Postgres MCP server to Claude Code
claude mcp add postgres -- npx -y @modelcontextprotocol/server-postgres \
  "postgresql://USER:PASSWORD@HOST:5432/your_database"

# Then restart Claude Code so it picks up the new server
🔌
Swap in the connector for your stack (Supabase, BigQuery, Snowflake…). The package name and credentials differ; the pattern — claude mcp add <name> -- <server command> — is the same.

Verify with one test query

Ask Claude to run a tiny query so you know the connection is live before going further:

prompt to Claude
Using my connected database MCP, list all table names,
then run: SELECT count(*) FROM <one of my tables>;
Show me the result.
✅
Check: you get back a real table list and a row count. If it errors, fix the connection now — the rest of the workflow depends on it.
3

Build schema.yml

Describe your tables so the workflow never guesses

schema.yml is the map of your database: every table, every column, what each one means, and how tables relate. The workflow treats it as ground truth — if a column isn't here, it doesn't exist as far as the analysis is concerned. Save your file at .claude/semantic-model/schema.yml.

The structure (top-level keys, in order)

KeyWhat it holds
versionSchema format version — keep 2.
models[]One entry per table: name, description, synonyms[], and columns[] (each column has name, description, data_type, synonyms[], optional constraints[]).
relationships[]How tables join: from_model/from_column → to_model/to_column + a description.
notesFree text: schema overview and any data-quality caveats.
common_business_questions[]FAQ-style questions with a ready query_pattern. Optional but high-value.
common_metrics[] / sql_style_guideQuick metric shortcuts and your SQL formatting rules. Optional.

Worked example — one real table

This is the accounts table from the workshop schema. Copy its shape exactly for your own tables — note how every column carries a plain-English description and synonyms (the words your team actually uses), so the agent maps questions to columns reliably.

.claude/semantic-model/schema.yml
version: 2

models:
  - name: accounts
    description: |
      Core accounts/organizations table. The top-level entity;
      users, subscriptions and events all hang off an account.
    synonyms: [organizations, companies, customers, tenants]
    columns:
      - name: id
        description: Unique identifier for the account (UUID)
        data_type: uuid
        synonyms: [account_id, org_id, company_id]
        constraints:
          - type: primary_key
          - type: not_null
      - name: industry
        description: Industry classification (e.g., SaaS, FinTech)
        data_type: text
        synonyms: [vertical, sector, business_type]
      - name: plan
        description: Subscription tier (e.g., Free, Pro, Enterprise)
        data_type: text
        synonyms: [tier, pricing_tier]
      - name: created_at
        description: When the account was created
        data_type: timestamp with time zone
        synonyms: [signup_date, onboarded_at]

relationships:
  - from_model: users
    from_column: org_id
    to_model: accounts
    to_column: id
    description: Each user belongs to exactly one account

notes: |
  Accounts are the top-level unit. Deletion cascades to users and events.
  Data-quality: most columns are nullable; standardize plan values.

Let Claude draft it — then you review

Don't hand-write the whole thing. Point Claude at your connected database and have it generate a first draft from the live schema, then you correct descriptions and add the synonyms your team really uses.

prompt to Claude
Inspect my connected database via MCP. For every table, read its
columns and data types. Generate a schema.yml that follows the EXACT
structure in .claude/semantic-model/ (version, models[] with
name/description/synonyms/columns, relationships[], notes).

For each table and column, write a plain-English description and a
few realistic synonyms. Infer relationships from foreign keys. Save
it to .claude/semantic-model/schema.yml, then show me anything you
were unsure about so I can correct it.
⚠️
You own the descriptions. Claude can read column names and types, but the meaning and synonyms come from your business. Review every description — this is the single biggest lever on answer quality.
4

Build metrics.yml

Define your metrics once, so they're computed consistently

metrics.yml is the dictionary of your business metrics — MRR, churn, active users, whatever matters to you. When someone asks for "churn", the workflow uses your definition here instead of inventing one. Save it at .claude/semantic-model/metrics.yml.

The structure

KeyWhat it holds
versionKeep 2.
metrics[]One entry per metric.
name / labelMachine name + human label.
descriptionOne sentence: what the metric means in business terms.
typesimple (one measure) or ratio (numerator ÷ denominator).
type_paramsFor simple: a measure. For ratio: numerator + denominator.
synonyms[]Every phrasing your team uses for this metric.
meta.agent_guidanceThe important one: plain-English instructions for how to compute it, plus source_table, filter, calculation pointers.

Worked examples — one simple, one ratio

.claude/semantic-model/metrics.yml
version: 2

metrics:
  # --- SIMPLE metric: one measure -------------------------
  - name: mrr
    label: "MRR"
    description: "Monthly Recurring Revenue: sum of monthly_price for active subscriptions."
    type: simple
    type_params:
      measure: mrr_sum
    synonyms: ["monthly revenue", "recurring revenue", "total billing"]
    meta:
      agent_guidance: "Sum 'monthly_price' from 'subscriptions' where status = 'active'."
      source_table: "subscriptions"
      source_column: "monthly_price"
      filter: "status = 'active'"

  # --- RATIO metric: numerator / denominator --------------
  - name: arpa
    label: "ARPA"
    description: "Average Revenue Per Account: average MRR per active account."
    type: ratio
    type_params:
      numerator: mrr_sum
      denominator: active_subscription_count
    synonyms: ["revenue per customer", "average deal size"]
    meta:
      agent_guidance: "Divide total MRR by the count of active subscriptions."
      source_table: "subscriptions"
      numerator_logic: "SUM(monthly_price) WHERE status = 'active'"
      denominator_logic: "COUNT(id) WHERE status = 'active'"

Let Claude draft it from plain language

Describe your metrics in everyday words; Claude turns them into the structured format, grounded in the schema.yml you just built.

prompt to Claude
Read my .claude/semantic-model/schema.yml. I'll describe my key
metrics in plain language and you write them into metrics.yml using
the EXACT structure in the workshop file (name, label, description,
type simple|ratio, type_params, synonyms, meta.agent_guidance with
source_table/filter/calculation).

My metrics:
- "Active customers" = count of accounts with an active subscription.
- "Churn rate" = canceled subscriptions / active subscriptions.
- "Weekly active users" = distinct users with an event in the last 7 days.
[...add yours...]

Save to .claude/semantic-model/metrics.yml and flag anything that
isn't computable from my schema.
⚠️
If a metric can't be computed from your schema, the workflow will say so and stop rather than guess — that's by design (see .claude/rules/no-guessing.md). Fix the schema or the metric definition, don't paper over it.
5

Write your business context

Teach the agent how to interpret your numbers

Numbers need context. The workshop keeps two short markdown files in .claude/business_knowledge/ — a company overview and a product catalog — that tell the agent what "good" looks like and how to read each customer segment. Replace them with yours, written in plain language.

business_desc.md

Who you are, who your customers are, the roles inside an account, how you package the product, and what a healthy account looks like.

products.md

Your SKUs/plans and how activity patterns differ across them — so "low usage" is judged correctly per product.

Let Claude draft it from a brain-dump

prompt to Claude
Here's a plain-language description of my company, customers, plans,
and what a healthy vs. at-risk account looks like:

[paste 1–2 paragraphs in your own words]

Rewrite this into two files matching the style and depth of the
workshop's .claude/business_knowledge/business_desc.md and
products.md. Keep it concrete and decision-oriented (what signals
matter, what "good" looks like per segment). Save both files into
.claude/business_knowledge/.
🎯
The best context says what to flag: e.g. "a quiet account with healthy API traffic is fine; a quiet account with falling API traffic is not." Those judgment cues are what make the agent's findings feel like an analyst wrote them.

Point the workflow at your local files

Swap Nimrod's GitHub source for your local context

Out of the box, the workflow loads schema.yml and metrics.yml from a GitHub repo via MCP. You'll redirect it to read the local files you just created. Two small edits.

Edit 1 — the semantic-model loader

Open .claude/skills/_shared/references/semantic-model-loader.md. Replace the "What to Load" GitHub instructions with local-file reads:

semantic-model-loader.md · before → after
# BEFORE — loads from GitHub via MCP
Tool: mcp__github__get_file_contents
  owner: "nimrodfisher"
  repo:  "workshop-queries-repo"
  path:  "schema.yml"

# AFTER — read the local files in this repo
Read .claude/semantic-model/schema.yml
Read .claude/semantic-model/metrics.yml
# If either file is missing, STOP and tell me — do not guess schema.

Edit 2 — the ground-truth pointer

In CLAUDE.md (and .claude/rules/ground-truth.md), the text says the semantic model is loaded from the connected GitHub repo. Update those pointers to say it's loaded from .claude/semantic-model/ locally, so the agent looks in the right place.

📝
You're only changing where the files are read from — not the rule itself. The principle is unchanged: schema and metrics are ground truth, never invented.
✅

Run an analysis

Pose a question and follow the 7 gated phases

You're set. Ask a real business question and let the pipeline run. It stops at a checkpoint after every phase to summarize findings and wait for your confirmation — so you always see what's happening and nothing runs away from you.

prompt to Claude
I want to run an analysis on my data. My question is:
"<your business question>"

Follow the full 7-phase workflow in CLAUDE.md. Load my local semantic
model first, scaffold an analysis folder, and stop at each checkpoint
for my confirmation before moving on.

What happens, phase by phase

#PhaseProduces
1hypothesis-framerTestable hypotheses + success criteria
2data-qaData-quality check before any analysis
3edaDistributions, segmentation, time series
4deep-analysisEffect sizes, driver decomposition
5synthesisFindings mapped back to each hypothesis
6validationSensitivity checks, alternative explanations
7data-storytellingInteractive HTML report + PDF summary
🧭
How to track where you are: each run scaffolds a folder under analyses/ with a plan.md whose checkpoint log updates after every phase. Open it any time to see what's done and what's next.
